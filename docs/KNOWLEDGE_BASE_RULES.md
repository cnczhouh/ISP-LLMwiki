# 知识库结构与规则

## 1. 设计目标

知识库结构优先服务于检索、引用、问答和长期维护，而不只是方便人工分类。

每条知识都应该尽量回答三个问题：

- 它来自哪里？
- 它适用于什么范围？
- 它应该如何被后续检索和引用？

## 2. 推荐目录

```text
Own/
  raw/                 # 来源层：原始资料
  wiki/                # 知识层：整理后的稳定知识
    modules/           # 模块页
    issues/            # 问题页
    workflows/         # 流程页
    tools/             # 工具页
    platforms/         # 平台页
    cases/             # 独立案例页，可选
    indexes/           # 索引页
  schema/              # 规则层：模板、命名规则、工作流
  logs/                # 过程层：摄取、重构、审查记录
```

### raw

`raw/` 是来源层，保留原始资料：

- 平台手册
- 应用手册
- datasheet
- tool 使用说明
- 会议记录
- 网页摘录
- 临时笔记
- 高质量 AI 回答草稿

规则：

- 不覆盖原始内容。
- 可做少量格式清理，但不改写原意。
- 文件名尽量体现主体、资料类型和标题。
- 若资料可能涉及版权或保密信息，上传 GitHub 前必须脱敏或排除。

### wiki

`wiki/` 是知识层，存放整理、抽象、关联后的稳定知识。

推荐分类：

- `wiki/modules/`：模块知识，例如 AE、AWB、Gamma、NR、HDR、Sharpness。
- `wiki/issues/`：问题知识，例如 偏色、噪声大、拖影、闪烁、高光过曝。
- `wiki/workflows/`：调试流程，例如 平台图像质量调整流程、跨平台风险检查表。
- `wiki/tools/`：工具使用，例如 ISP Tuning Tool、寄存器访问。
- `wiki/platforms/`：平台主体，例如 ISX031、SC121AT、GW5、FH833X。
- `wiki/cases/`：独立案例，适合较完整的项目复盘。
- `wiki/indexes/`：导航和聚合索引。

### schema

`schema/` 是规则层，适合存放：

- 页面模板
- 命名规则
- 入库工作流
- Lint 审查规则
- 原始资料模板

### logs

`logs/` 是过程层，只记录维护过程，不承载正式结论。

适合记录：

- 本次处理了哪些原始资料。
- 新建、合并、重命名或删除了哪些页面。
- 更新了哪些索引。
- 有哪些判断需要人工确认。
- Lint 审查发现和处理结果。

## 3. 页面角色

新增或更新页面前，先判断页面角色：

| 页面角色 | 目录 | 适合内容 |
| --- | --- | --- |
| 通用模块 | `wiki/modules/` | 跨平台成立的模块原理、参数作用、调试取舍 |
| 通用问题 | `wiki/issues/` | 现象、可能原因、排查顺序、典型误判 |
| 通用流程 | `wiki/workflows/` | 某类问题或平台 bring-up 的调试路径 |
| 通用工具 | `wiki/tools/` | 工具入口、操作步骤、观察指标、常见误用 |
| 平台页 | `wiki/platforms/` | 某 sensor / ISP / 方案 / 厂家的平台知识 |
| 案例页 | `wiki/cases/` 或问题页小节 | 真实项目过程、参数修改、最终结论 |
| 索引页 | `wiki/indexes/` | 导航、聚合、入口列表 |

## 4. 页面属性

页面开头建议显式写出页面属性，帮助模型判断页面角色。

推荐字段：

```markdown
## 页面属性
- 类型：通用模块 / 通用问题 / 通用流程 / 通用工具 / 平台 / 案例 / 索引
- 主类型：Sensor / ISP / Solution / Vendor
- 附加属性：集成 ISP / HDR / 前级补偿 / 车载 / 低照优化
- 厂家：
- 平台：
- 模块：
- 场景：
- 适用范围：跨平台 / 指定平台 / 指定方案
```

说明：

- 不强求所有字段都有值。
- 混合对象使用“主类型 + 附加属性”，不要硬塞进单一分类。
- 例如 ISX031 可写成主类型 Sensor，附加属性为集成 ISP / HDR / 前级补偿。

## 5. 内容放置原则

### 通用知识优先写主干

模块页、问题页、流程页应先写跨平台主干：

- 模块作用
- 调试目标
- 关键观察项
- 典型调试方向
- 参数联动关系
- 常见副作用

平台差异可以作为入口补充，但不要让单一平台实现淹没通用逻辑。

### 平台知识写平台页

平台页适合写：

- 平台定位
- 厂家归属
- 支持的 ISP 模块
- 工具入口
- 平台特有参数或页面
- 常见问题映射
- 与通用问题、模块、流程的链接

### 项目经验写成案例

真实项目经验优先放在对应问题页的“调试案例 / 项目记录”小节，或写成 `wiki/cases/` 独立案例页。

规则：

- 保留场景、现象、排查过程和最终结论。
- 不把一次经验伪装成无条件通用规律。
- 若横跨多个问题，选择一个主问题完整记录，其他页面只做链接引用。

## 6. 命名规则

### raw 文件

建议格式：

```text
主体_资料类型_标题.pdf
主体_资料类型_标题.md
```

示例：

```text
ISX031_datasheet_E_E23406_COB.pdf
SC121AT_ISP_Tuning_Tool使用指南_V1.2.pdf
GW5_ISP_Tuning_LTM.pdf
```

### wiki 文件

建议：

- 平台页：`ISX031.md`、`SC121AT.md`、`GW5.md`
- 平台模块页：`ISX031_AE.md`、`GW5_LTM.md`
- 通用模块页：`AE.md`、`AWB.md`、`NR.md`
- 问题页：`偏色.md`、`噪声大.md`、`拖影.md`
- 流程页：`SC121AT_图像质量调整流程.md`
- 索引页：`ISP调试知识库导航.md`、`ISX031_索引.md`

原则：

- 单一平台资料整理结果，模块页建议带平台前缀。
- 已抽象为跨平台共性的页面，可使用不带平台前缀的通用名称。
- 避免重复创建同义页面，例如“偏黄”优先并入“偏色”。

## 7. 入库工作流

1. 把原始资料放入 `raw/`。
2. 识别厂家、主体对象、模块、问题、工具、流程和可沉淀案例。
3. 判断资料更适合进入通用页、平台页、问题案例，还是索引页。
4. 给页面补齐页面属性。
5. 先抽取主干答案，再补平台入口。
6. 建立双向链接，至少关联一个平台或主体、一个模块/问题/流程、一个来源。
7. 追加维护日志，记录处理范围和待确认判断。

## 8. 链接与引用

统一使用 Obsidian 风格链接：

```markdown
[[AWB]]
[[偏色]]
[[wiki/modules/GW5_LTM|GW5_LTM]]
```

重要结论必须能追溯到：

- `raw/` 中的原始资料
- `wiki/cases/` 或问题页中的真实案例
- 维护日志中记录的整理过程

## 9. 面向 RAG 的写法

为了让大模型检索更稳定：

- 页面标题要明确，不要只写缩写。
- 小节标题要稳定，例如“调试视角”“关键参数”“排查顺序”“来源”。
- 每个页面开头先说明“这一页是什么”。
- 表格适合整理参数、目标、收益和副作用。
- 结论不要脱离适用范围。
- 索引页只负责导航，不承载大量细节结论。

## 10. GitHub 公开注意事项

- 不上传密钥、 `.env`、`.env.local`。
- 不上传无权公开的 PDF、datasheet、客户项目记录或厂商保密资料。
- 如果要展示完整结构，可放少量脱敏示例页。
- 可以把 `raw/` 目录保留为空目录或仅放公开资料。
- README 中的本地路径应改成示例路径，真实路径放在 `.env.local`。
