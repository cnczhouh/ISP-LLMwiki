# 知识架构说明

## 目标
知识库结构优先服务于后续大模型检索、引用和问答，而不只是方便人工分类。

## 推荐目录
保持简单主目录：
- `raw/`
- `wiki/modules/`
- `wiki/issues/`
- `wiki/workflows/`
- `wiki/tools/`
- `wiki/platforms/`
- `wiki/indexes/`
- `schema/`
- `logs/`

说明：
- `raw/` 是来源层，保留原始资料
- `wiki/` 是知识层，存放整理后的稳定知识
- `schema/` 是规则层，存放模板、命名规则和工作流
- `logs/` 是过程层，记录摄取、更新、重构和 Lint 审查
- 不依赖深目录表达细分类
- 更细的分类信息优先通过页面属性表达
- 页面之间通过双向链接和索引聚合建立关系

## 维护层：logs
`logs/` 不承载正式知识结论，只记录知识库维护过程。

适合记录：
- 从 `raw/` 摄取资料进入 `wiki/` 的过程
- 新建、合并、重命名、重构页面的原因
- 索引页更新范围
- Lint 审查发现的问题、已处理项和待确认项

说明：
- 稳定结论仍应写入 `wiki/`
- 原始依据仍应保留在 `raw/`，真实项目过程应沉淀到对应 `wiki/issues/` 问题页
- 日志按月组织即可，例如 `logs/2026-04.md`

## 页面属性优先
建议在页面头部显式写出页面属性，帮助模型判断页面角色。

推荐字段：
- 类型：通用模块 / 通用问题 / 通用流程 / 通用工具 / 平台 / 案例 / 索引
- 主类型：Sensor / ISP / Solution / Vendor
- 附加属性：集成 ISP / HDR / 前级补偿 / 车载 / 低照优化 等
- 厂家：
- 平台：
- 模块：
- 场景：
- 适用范围：跨平台 / 指定平台 / 指定方案

## 内容放置原则
### 1. 通用知识
适合放入：
- `wiki/modules/`
- `wiki/issues/`
- `wiki/workflows/`
- `wiki/tools/`

说明：
- 写跨平台仍成立的模块原理、问题排查、流程方法、工具使用方式
- 页面中可保留“平台差异入口”，但主干答案应先保持通用

### 2. 平台知识
适合放入：
- `wiki/platforms/`

说明：
- 写某个 sensor、某个 ISP、某套方案或某厂家主体知识
- 不强求严格单选归类，用“主类型 + 附加属性”表达更准确

### 3. 实战经验
适合放入：
- 对应的 `wiki/issues/` 问题页

说明：
- 真实项目、真实调试过程、真实结论优先写到问题页的“调试案例 / 项目记录”小节
- 不要把案例经验直接上升为无条件通用规律
- 若某个案例横跨多个问题，可在主要问题页完整记录，在其他问题页只做链接引用

## 主归类 + 附加属性
对于混合型对象，不强制严格单选。

例如：
- `ISX031`
  - 主类型：Sensor
  - 附加属性：集成 ISP / HDR / 前级补偿
- 某独立 ISP
  - 主类型：ISP
  - 附加属性：自动控制链路 / 图像风格链路 / 独立调试工具

## 模块页命名补充规则
- 若模块页主要是单一平台资料整理结果，文件名建议带平台前缀，例如 `ISX031_AE.md`
- 若模块页已经抽象成跨平台共性知识，再使用不带平台前缀的通用模块名，例如 `AE.md`
- 这样可以同时兼顾目录辨识度与后续跨平台扩展

## 页面放置判断标准
新增知识时优先问自己：
1. 这是不是跨平台都成立？
2. 这是不是某个平台主体的知识？
3. 这是不是某次项目经验？它主要对应哪个问题现象？

按答案决定放入通用页、平台页，或沉淀到对应问题页的案例小节。
